## Requirement 1: ML Model Documentation

### Status: PARTIALLY MET

### What exists
- `ML_FEATURES_DOCUMENTATION.md` — feature list for GBDT, GNN, LSTM
- `ARCHITECTURE.md` — describes each model's role in the pipeline
- `MODEL_PERSISTENCE_IMPLEMENTATION.md` — covers model persistence and metadata
- `TEMPORAL_SUBSYSTEM.md` — deep dive on LSTM link predictor
- `src/gbdt_detector.py`, `src/gnn_trainer.py`, `src/lstm_link_predictor.py` — source with inline comments
- Model metadata JSON files (stored alongside `.pt`/`.txt` model files in `models/`)

### What is missing

| Gap | Detail |
|-----|--------|
| **Training data source** | All training data is synthetic (generated by `transaction_simulator.py`). There is no documentation explaining *why* synthetic data was chosen, how it mirrors real transaction distributions, or what real datasets (e.g., IEEE-CIS, Elliptic, PaySim) were considered or rejected. |
| **Validation methodology** | Models use a single train/test split per run. There is no k-fold cross-validation, no confidence intervals, and no documentation of the validation strategy rationale. |
| **Benchmark datasets** | No comparison against public AML/fraud detection benchmarks. No external dataset was used to validate whether synthetic-trained models generalise to real transaction patterns. |
| **Data distribution documentation** | Missing analysis of class imbalance ratios used, feature distributions, and how these compare to real-world AML data (typical fraud rate: 0.1–1%). |
| **Formal technical report** | No single consolidated document that a regulator or client reviewer could read to understand model governance. Existing docs are developer-focused, not auditor-focused. |

### Work needed
1. Write a **Model Governance Document** covering:
   - Each model's purpose, architecture, training data source, and known limitations
   - Why synthetic data was chosen and its fidelity constraints
   - Validation split strategy and reproducibility (random seed, split ratio)
   - Comparison to at least one published benchmark (even if only directional)
2. Add cross-validation (k=5) to `train.py` and record per-fold metrics
3. Document hyperparameter choices with justification (current values are hard-coded with no rationale)

## Requirement 2: 90%+ Accuracy Claim — Full Test Harness Report

### Status: NOT MET

### What exists
- `evaluate_class0.py` — evaluates false positive rate on Class 0 (legitimate) nodes only
- `test_dashboard_system.py` — 7-point system smoke test (connectivity, model loading, endpoint reachability)
- `inference_test_results.csv` — per-account scoring results (scores + risk levels, no ground-truth labels)
- `scripts/test_gnn_integration.py` — GNN pipeline integration test
- GBDT internally logs accuracy metrics during training (stdout only, not persisted to a report)

### What is missing

| Gap | Detail |
|-----|--------|
| **No ground-truth test dataset** | `inference_test_results.csv` contains scores but no labelled ground truth to compute precision/recall/F1 against. There is no held-out labelled test set. |
| **No precision/recall/F1 report** | Raw classification metrics (TP, FP, FN, TN) are not persisted anywhere. The 90%+ accuracy claim has no supporting artefact. |
| **No ROC / PR curves** | No visualisations of classifier performance across thresholds. |
| **No per-model test report** | GBDT, GNN, LSTM, Sequence Detector each need individual test results — not just a combined pipeline score. |
| **No formal test harness** | `test_dashboard_system.py` is an infrastructure smoke test, not a machine-learning evaluation harness. There is no pytest suite that loads models and scores a labelled dataset. |
| **Accuracy definition unclear** | "90%+ accuracy" is ambiguous — overall accuracy on a class-imbalanced dataset is misleading. Captivus will need F1, AUROC, and precision at a given recall threshold. |

### Work needed
1. Create a **labelled evaluation dataset** from the simulator with known fraud labels retained
2. Write a **`evaluate_models.py` test harness** that:
   - Loads each model
   - Scores the labelled test set
   - Outputs a report with: accuracy, precision, recall, F1, AUROC, confusion matrix — per model and combined
3. Save the report as `MODEL_EVALUATION_REPORT.md` and raw numbers to `evaluation_results.json`
4. Generate ROC and Precision-Recall curve plots (save to `outputs/`)
5. Clarify the accuracy definition in all documentation — replace "90%+ accuracy" with the specific metric and threshold

## Requirement 3: Scalability — Transactions Per Second at Peak Load

### Status: NOT MET

### What exists
- `ARCHITECTURE.md` — theoretical complexity estimates (e.g. "GBDT batch scoring: 10–20 ms/batch")
- `pipeline_simulation.py` — runs the full pipeline and records execution time against a synthetic dataset
- `simulation_pipeline_results.csv` — contains pipeline run metrics (timing, counts)
- Memory footprint documented: ~1.3 GB models loaded, ~20 MB/100k transactions working graph

### What is missing

| Gap | Detail |
|-----|--------|
| **No TPS figure** | There is no measurement of transactions per second. The architecture doc gives phase-level latency but not end-to-end throughput under sustained load. |
| **No load test** | No tool (Locust, k6, JMeter, custom script) has been used to hammer the API or pipeline under concurrent load. |
| **No concurrency testing** | The Flask API has not been tested under simultaneous requests. Race conditions or memory exhaustion at scale are unknown. |
| **No peak load definition** | What constitutes "peak load" for this system? (e.g. 10k txs/min, 1M txs/hour?) This has not been specified or tested. |
| **No scalability limits documented** | The cycle detection bounds (`max_k=6`, `max_cycles=500`) are noted but no analysis exists of how accuracy degrades as these limits are reached. |
| **Single-node only** | No distributed processing. Horizontal scaling strategy is absent. |

### Work needed
1. Define **peak load target** (e.g. "10,000 transactions per minute" or SLA from Captivus)
2. Write a **load test script** using Locust or k6 targeting:
   - `/consolidate/risks` endpoint
   - `/score/transaction` endpoint
   - Simulate concurrent users at 1x, 5x, 10x expected load
3. Measure and document:
   - Median, p95, p99 response latency (ms)
   - Maximum sustainable TPS before error rate exceeds 1%
   - Memory and CPU behaviour under sustained load
4. Produce a **Load Test Report** (`LOAD_TEST_REPORT.md`) with methodology, results, and bottleneck analysis
5. Document scale-out strategy (even if only theoretical for now)

## Requirement 4: Security Architecture — Encryption, Access Controls, Audit Trails

### Status: NOT MET

### What exists
- Node IDs are anonymised (no raw PII stored in embeddings)
- `metrics.db` (SQLite) logs inference requests with timestamp and account ID
- `configs/rules.yml` — configuration toggles for detection patterns
- Adversarial training hardens models against evasion (model robustness, not data security)

### What is missing

| Gap | Detail |
|-----|--------|
| **No authentication** | The REST API (`/score/transaction`, `/consolidate/risks`, etc.) has zero authentication. Any client with network access can call all endpoints. |
| **No authorisation / RBAC** | No concept of roles (analyst vs. admin vs. read-only). All users have identical access. |
| **No encryption at rest** | Model files, SQLite databases, and CSV outputs are stored as plaintext on disk. |
| **No encryption in transit** | The Flask API serves HTTP, not HTTPS. Credentials and transaction data travel unencrypted. |
| **Incomplete audit trail** | `metrics.db` logs scores but does not record: who made the request, what input was submitted, what action was taken downstream. |
| **No immutable audit log** | Logs are in a mutable SQLite file. An auditor or regulator will require append-only, tamper-evident logging. |
| **No secrets management** | No `.env` support, no vault integration. Configuration is in plain YAML/Python defaults. |
| **No input validation** | API endpoints accept arbitrary JSON. Malformed or malicious payloads are not rejected before processing. |
| **No rate limiting** | The API has no throttling. It is vulnerable to abuse or denial-of-service. |
| **No security architecture document** | There is no document explaining the security model, threat surface, or controls — required for a financial services client. |

### Work needed
1. **Authentication**: Add API key authentication (minimum) or JWT to all Flask endpoints
2. **HTTPS**: Configure TLS termination (nginx reverse proxy or Flask-Talisman)
3. **Input validation**: Add Pydantic models to all API request schemas
4. **Audit logging**: Extend `metrics_logger.py` to record requester identity, full input hash, output, and timestamp — store in append-only format
5. **Rate limiting**: Add Flask-Limiter with per-IP and per-key limits
6. **Encryption at rest**: Document or implement encryption for model files and databases (at minimum, document the host-level encryption expectation)
7. **Write `SECURITY_ARCHITECTURE.md`** covering:
   - Threat model (who could attack, what they could do)
   - Controls in place (authentication, encryption, validation)
   - Controls planned but not yet implemented
   - Compliance posture (GDPR data minimisation, AML record-keeping obligations)
