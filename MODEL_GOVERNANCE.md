## 1. Overview

This document provides auditor-grade governance documentation for all machine learning models deployed in the AML Topology Detection System. It covers training data provenance, validation methodology, hyperparameter choices, known limitations, and recommended real-data validation steps.

### Models Covered

| Model | Type | Primary Role |
|-------|------|--------------|
| GBDT (LightGBM) | Gradient Boosted Decision Tree | Transaction-level fraud scoring |
| GNN Node Classifier | Graph Neural Network | Network-level suspicious node classification |
| LSTM Link Predictor | Recurrent Neural Network | Temporal link prediction between accounts |
| Sequence Detector | LSTM Autoencoder | Event-sequence anomaly detection |
| PPO Adversarial Agent | Reinforcement Learning (PPO) | Red-team evasion pattern generation |

## 2. Training Data

### 2.1 Source

All models are trained on **synthetic data** generated by `src/simulator.py` and `src/gbdt_detector.py`. No real customer transaction data is used in the current version.

### 2.2 Rationale for Synthetic Data

Real labelled AML transaction datasets are not publicly available at the required scale. Synthetic data allows:
- Controlled injection of known FATF fraud typologies (fan-in, fan-out, layering cycles)
- Reproducible benchmarking with fixed random seeds
- Risk-free experimentation without exposing customer PII

### 2.3 Fidelity — What It Approximates

| Real-World Distribution | Synthetic Approximation |
|-------------------------|------------------------|
| Transaction amounts | `np.random.exponential(mean=500)` — allows realistic high-value transactions |
| Account activity frequency | Zipf-like via modulo-bucket assignment |
| Fraud rate | ~10% (feature-derived; real AML typically 0.1–1%) |
| Fraud typologies | Feature-correlated risk scoring: high amount, high IP risk, device change, crypto, high-risk country, velocity |
| Geographic risk | High-risk country codes (RU, NG, PK, CN) weighted at ~15% of traffic |
| Device/IP risk | `np.random.beta(2,5)` — skewed toward low risk, matching real distributions |

### 2.4 Evaluation Dataset

A fixed held-out test set was generated for Phase 2 evaluation with the following properties:

| Property | Value |
|----------|-------|
| Total samples | 50,000 |
| Held-out test set | 10,001 |
| Training context | 39,999 |
| Random seed | 2025 (fixed) |
| Fraud rate (test) | 9.65% (965 fraud cases) |
| Generation date | 2026-04-05 |
| CSV SHA-256 | `e93eabb588a8975c79794f825e8f7673ebb00a4616af2666e8f1d68d55d61a80` |
| Metadata file | `data/eval_dataset_metadata.json` |

### 2.5 Comparison to Public Datasets

| Dataset | Methodology | Similarity |
|---------|-------------|------------|
| PaySim (Lopez-Rojas, 2016) | Synthetic mobile money simulation | **Most similar** — same synthetic labelling approach |
| Elliptic Bitcoin Dataset | Real-world graph transactions | **Closest analogue** for GNN — real labels but crypto-specific |
| IEEE-CIS Fraud Detection | Card-not-present e-commerce | Different domain; no graph structure |

### 2.6 Data Limitations

- Synthetic data does not capture the long-tail complexity of real financial crime schemes
- Class imbalance (3.3%) is higher than typical real-world AML rates (0.1–1%), inflating recall estimates
- Account relationship graphs are randomly generated; real-world typologies exhibit denser community structure
- The GBDT featurisation uses session-level velocity counters that may behave differently on real streaming data

## 4. Accuracy Claim Assessment

### 4.1 The "90%+" Claim

The system marketing references "90%+ accuracy." This section provides an honest interpretation of what the metrics support.

**Overall accuracy ≥ 90% is supported** for GBDT and Combined Pipeline (96.8–96.9%) — but this is a misleading metric at 3.3% fraud rate. A trivial classifier that labels everything as legitimate achieves 96.7% accuracy. Overall accuracy should not be cited as a performance indicator for this system.

**The meaningful metrics are F1 score and AUROC:**

| Claim | Supported by Data? | Correct Metric |
|-------|--------------------|----------------|
| "90% accuracy" (overall) | Partial — held-out accuracy = 88.97% (below 90%) | Overall accuracy is misleading at ~10% fraud rate; cite F1 and AUROC instead |
| "47%+ F1 on transactions" | **Yes** — held-out GBDT F1 = 47.3%; CV F1 = 44.8% | Genuine discriminative performance on feature-correlated synthetic data |
| "88%+ AUROC on transactions" | **Yes** — held-out GBDT AUROC = 87.99%; CV AUROC = 86.2% | Strong discrimination well above random baseline (50%) |
| "90% F1 on event sequences" | Yes — Sequence Detector F1 ≈ 100% | Valid; caveat: in-distribution synthetic test |
| "90% F1 on link prediction" | Yes — LSTM F1 = 98.9% | Valid; caveat: small test set (600 pairs) |
| "90% F1 on node classification" | Yes — GNN F1 = 91.2% | Valid; caveat: small test set (101 nodes), high fraud prevalence (82%) |

**Recommended client-facing framing:**
> "The transaction-level GBDT achieves AUROC 0.88 and F1 0.47 on the held-out synthetic test set (10,001 transactions, 965 fraud cases). The sequence anomaly and link prediction components achieve F1 > 95%. All metrics are derived from synthetic data with feature-correlated fraud labels. A 90-day pilot with labelled historical data from Captivus is required before production accuracy claims can be substantiated."

## 5. Hyperparameter Choices

### 5.1 GBDT (LightGBM)

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `num_boost_round` | 500 | More rounds with lower LR for better convergence |
| `num_leaves` | 63 | Deeper trees to capture feature interactions |
| `learning_rate` | 0.03 | Slower LR reduces overfitting on minority class |
| `objective` | `binary` | Binary classification task |
| `metric` | `auc` | Optimise AUROC directly; more meaningful than logloss under imbalance |
| `scale_pos_weight` | neg/pos ratio | Compensates for class imbalance |
| `min_child_samples` | 20 | Prevents overfitting on minority-class leaf nodes |
| `feature_fraction` | 0.8 | Column subsampling for regularisation |
| `bagging_fraction` | 0.8 | Row subsampling for regularisation |
| `reg_alpha` / `reg_lambda` | 0.1 / 0.1 | L1/L2 regularisation |
| Classification threshold | F1-optimal (per fold) | Sweeps 0.05–0.95 on validation set; avoids fixed-0.5 bias |
| Features | 21 engineered features | See `src/gbdt_detector.py:featurize()` |

**New features vs v1.0** (added in v1.1):

| Feature | Type | Rationale |
|---------|------|-----------|
| `amt_gt_1k`, `amt_gt_3k` | Binary | Non-linear amount thresholds matching fraud triggers |
| `is_crypto` | Binary | Direct payment-method risk flag |
| `is_high_risk_country` | Binary | Explicit high-risk geography (RU, NG, PK, CN) |
| `ip_risk_high`, `ip_risk_mid` | Binary | Bucketised IP risk for non-linear GBDT splits |
| `high_velocity` | Binary | Velocity spike flag (`count_1h > 8`) |
| `amt_x_ip_risk` | Interaction | Amount × IP risk — key fraud interaction |
| `high_amt_device_change` | Interaction | High amount AND device change together |
| `crypto_high_risk_country` | Interaction | Crypto payment from high-risk jurisdiction |

### 5.2 GNN Node Classifier

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `hidden_dim` | 64 | Sufficient capacity for 12 input node features |
| `num_layers` | 2 | 2-hop neighbourhood — captures direct counterparties |
| `dropout` | 0.3 | Regularisation for small synthetic graphs |
| Aggregation | Mean pooling | Stable under varying node degrees |

### 5.3 LSTM Link Predictor

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `hidden_size` | 64–128 | Standard for sequence tasks of this length |
| `num_layers` | 1–2 | Empirically stable; deeper layers did not improve validation AUC |
| `seq_len` | 3 time steps | Minimum temporal context (12H frequency → 36H window) |
| `dropout` | 0.1 | Light regularisation for short sequences |
| `learning_rate` | 0.001 (Adam) | Standard for LSTM training |

### 5.4 Sequence Detector (LSTM Autoencoder)

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `seq_len` | 15 events | Captures typical transaction session length |
| `hidden_size` | 64 | Sufficient for event vocabulary size |
| `epochs` | 5 | Short training; reconstruction loss converges quickly on synthetic data |
| Threshold | 0.86 (auto-tuned) | Selected to maximise F1 on held-out validation |

### 5.5 PPO Adversarial Agent

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| `state_dim` | 49 | Graph summary statistics + GNN embedding |
| `hidden_dim` | 128 | Standard PPO actor-critic capacity |
| `learning_rate` | 3e-4 | Schulman et al. (2017) recommended default |
| `episodes` | 100 (default) | Sufficient for convergence on small synthetic graphs |
| `steps_per_episode` | 10 | Pattern injection budget per episode |

## 6. Known Limitations

| Limitation | Impact | Affected Models |
|------------|--------|-----------------|
| **Synthetic training data** — real-world distribution shift unknown | High — F1 may degrade significantly on real data | All |
| **Synthetic fraud labels** — derived from heuristic risk score, not real SAR filings | High — label noise differs from real fraud patterns | GBDT |
| **Class imbalance** — ~10% synthetic fraud rate is higher than real AML (~0.1–1%) | Medium — precision/recall trade-off calibrated incorrectly; real-data threshold re-tuning required | GBDT, Combined Pipeline |
| **Cold-start problem** — LSTM requires historical sequence data | Medium — new accounts cannot be scored temporally | LSTM Link Predictor |
| **Minimum history requirement** — velocity features need ≥1 day history | Low — affects first-day transactions | GBDT |
| **Cycle detection bound** — `max_k=6` hops | Medium — deep layering chains (>6 hops) evade graph analysis | GNN, Cycle Detector |
| **Sequence Detector overfit risk** — near-100% metrics on synthetic in-distribution test | High — cannot rely on these metrics for real deployment | Sequence Detector |
| **GNN test set size** — 101 nodes, 83 fraud (82% prevalence) | High — prevalence far exceeds real-world rates; metrics are unreliable | GNN Node Classifier |
| **No temporal holdout** — CV folds are not time-ordered | Medium — may overestimate performance due to temporal leakage | GBDT |
| **Feature stability** — MCC and payment type encodings are fit on training data | Low — OOV categories mapped to -1 fallback | GBDT |
| **Threshold portability** — F1-optimal threshold tuned on synthetic data | Medium — threshold must be re-derived on real labelled data before deployment | GBDT |

## 7. Recommended Real-Data Validation Steps

### Phase 1: Data Acquisition
1. Obtain 3 months of labelled historical transactions from Captivus (minimum 100,000 transactions; minimum 1,000 confirmed SAR cases)
2. Ensure labels are SAR-filing outcomes, not rule-based flags
3. Agree on a temporal holdout: train on months 1–2, test on month 3

### Phase 2: GBDT Re-training
1. Re-run `scripts/generate_eval_dataset.py` style pipeline on real data
2. Retrain GBDT with real labels; the 21 engineered features in v1.1 are a validated starting point
3. Add new features available in real data (e.g. KYC tier, account age, beneficial ownership flag)
4. Re-derive the F1-optimal classification threshold on real validation data (do not carry over the synthetic threshold)
5. Re-run 5-fold temporal CV; target F1 ≥ 0.45 and AUROC ≥ 0.80 as production gates (aligned with synthetic baseline)

### Phase 3: GNN and LSTM Fine-Tuning
1. Build real transaction graph; freeze GNN topology weights from synthetic pre-training
2. Fine-tune GNN classifier head on real labelled nodes (2-epoch fine-tune recommended)
3. Re-evaluate LSTM on real account sequences using the same `evaluate_models.py` harness

### Phase 4: Re-run Governance Documentation
1. Re-execute `evaluate_models.py` to regenerate `MODEL_EVALUATION_REPORT.md` with real metrics
2. Update this document's Section 3 and 4 with production metric values
3. Obtain sign-off from compliance and data science leads before production deployment
